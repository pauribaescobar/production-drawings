# SolidEdgeCommunity.Reader Worker

Windows-only proof of concept for converting a Solid Edge Draft (`.dft`) into per-sheet EMF files, overlaying business-data annotations from the parsed order JSON, and finalizing a PDF using managed PDF generation.

## End-to-end flow

1. `Program.cs` parses `export <input.dft> <output-directory> <parsed-order.json> [job-id]`.
2. `DraftConversionPipeline.cs` coordinates the run.
3. `DraftEmfExporter.cs` validates the input, opens the draft with `SolidEdge.Community.Reader`, and writes `sheet-001.emf`, `sheet-002.emf`, and so on.
4. `ParsedOrderDocument.cs` loads the parsed order JSON and exposes the order-level fields used in the annotation block.
5. `SheetAnnotation.cs` builds one annotation payload per sheet from the parsed order data.
6. `MicrosoftPrintToPdfWriter.cs` renders each EMF page, overlays the treatment, dimension values, and value-only business fields on top of the original drawing in the available free space, and writes the final PDF as `final.pdf` using managed PDF generation.
7. `Program.cs` emits a JSON worker result to stdout.

## Files in this experiment

- `SolidEdgeReaderPoc.csproj`: console app targeting `net48` with `SolidEdge.Community.Reader` `2.0.0`.
- `Program.cs`: CLI entry point, exit code handling, and JSON emission.
- `DraftConversionPipeline.cs`: top-level orchestration for export -> annotate -> PDF.
- `DraftEmfExporter.cs`: DFT-to-EMF adapter using `DraftDocument.Sheets`.
- `MicrosoftPrintToPdfWriter.cs`: Windows-only PDF finalization and annotation overlay renderer using managed PDF generation.
- `ParsedOrderDocument.cs`: lightweight JSON loader for the parsed order payload.
- `SheetAnnotation.cs`: per-sheet metadata record derived from the parsed order.
- `ExportResult.cs`: intermediate result model containing input/output paths, sheet count, EMF paths, and final PDF path.
- `WorkerResult.cs`: stable success/failure payload returned to the caller.
- `WorkerJson.cs`: hand-rolled JSON serializer for the worker contract.
- `DraftPdfFinalizationException.cs`: signals Windows PDF finalization failures separately from export failures.
- `IDraftConversionPipeline.cs`, `IDraftEmfExporter.cs`, `IAnnotatedPdfWriter.cs`: seams for testability and replacement.
- `.gitignore`: excludes `bin/`, `obj/`, and common IDE user files.

## Worker contract

Invocation:

```bash
SolidedgeReaderPoc export <input.dft> <output-directory> <parsed-order.json> [job-id]
```

Inputs:

- `input.dft`: a single Solid Edge Draft file.
- `output-directory`: writable directory for EMFs and `final.pdf`.
- `parsed-order.json`: parsed order payload generated by the web app flow.
- `job-id`: optional correlation id; autogenerated when omitted.

Outputs:

- JSON on stdout.
- Exit code `0` on success.
- Exit code `1` for usage errors.
- Exit code `2` for export or finalization failures.
- `errorCode` values currently used: `usage_error`, `export_error`, `pdf_finalization_error`.

Success payload shape:

```json
{
  "status": "succeeded",
  "jobId": "abc123",
  "inputPath": "C:\\input.dft",
  "outputDirectory": "C:\\output",
  "sheetCount": 3,
  "artifacts": [
    "C:\\output\\sheet-001.emf",
    "C:\\output\\sheet-002.emf",
    "C:\\output\\sheet-003.emf",
    "C:\\output\\final.pdf"
  ],
  "errorCode": null,
  "errorMessage": null
}
```

Failure payload shape:

```json
{
  "status": "failed",
  "jobId": "abc123",
  "inputPath": "C:\\input.dft",
  "outputDirectory": "C:\\output",
  "sheetCount": 0,
  "artifacts": [],
  "errorCode": "export_error",
  "errorMessage": "Draft file not found."
}
```

## Assumptions and caveats

- Windows only.
- Requires a Windows-capable .NET runtime that can reference the archived reader package.
- Requires `SolidEdge.Community.Reader` `2.0.0`.
- Requires the output directory to be writable.
- Requires the parsed order JSON generated by the matching web app flow.
- Requires the parsed order JSON line reference to match the DFT file name for each exported sheet.
- Uses `PdfSharp` for the final PDF output and does not require the Windows print subsystem. The generated PDF layout keeps the original drawing visible, overlays the resolved values without a detached right-side panel, and does not redraw placeholder field labels.
- Input validation is limited to path checks, file existence, and `.dft` extension validation before the reader opens the file.
- Partial artifacts are not cleaned up if the pipeline fails midway.
- JSON serialization is custom, so the response contract is maintained manually.
